# Спека: редактирование md-файлов — слайс 7

Дата: 2026-07-08 · Tracker: none (key-less)
Опирается на: просмотр (VaultRepository/MarkdownScreen) и синк-слайсы.

## Цель
Дать править содержимое открытой заметки и сохранять на диск (в тот же vault). Сохранённые
изменения подхватываются существующим git-синком (кнопка «Синхронизировать» / автосинк).

## Границы (scope)
**В слайсе:** `VaultRepository.writeFile`, `VaultViewModel.saveFile`, режим правки в
`MarkdownScreen` (просмотр ↔ редактор).
**НЕ в слайсе:** создание нового файла, переименование, удаление, live-preview, авто-сохранение,
запуск синка по сохранению.

## Проектные решения
- Правка — только открытого существующего файла. Режим-переключатель на экране заметки
  (не отдельный экран).
- Сохранение = запись файла; синк отдельно (кнопка/автосинк). Не дублируем триггер (YAGNI).
- Локальное состояние режима (правка/просмотр) и черновика — в `MarkdownScreen` (`remember`);
  `VaultViewModel` только персистит и обновляет `content`.

## Архитектура и интерфейсы

### VaultRepository (commonMain) — добавить запись
```kotlin
fun writeFile(path: String, content: String)   // okio: создать/перезаписать
```
Дополняет существующие `listMarkdownFiles()` / `readFile(path)`.

### VaultViewModel (commonMain) — сохранение
```kotlin
fun saveFile(path: String, content: String)   // repo.writeFile(...); _state.content = content
```
Пишет на IO-диспатчере (как чтение), затем обновляет `state.content`.

### MarkdownScreen (commonMain, `app.obsidianmd.ui`)
Режим-переключатель (локальный `remember`):
- **Просмотр:** отрендеренный markdown (как сейчас) + кнопка «Редактировать» + «Назад».
- **Редактирование:** `OutlinedTextField` с сырым текстом (инициализируется из `content`) +
  «Сохранить» (→ `onSave(text)`, возврат в просмотр) + «Отмена» (возврат без записи).
Сигнатура: `MarkdownScreen(content, onBack, onSave)`; `onSave` вызывает `vm.saveFile(selected.path, text)`.

   == "# Hi"; перезапись существующего файла новым текстом → `readFile` возвращает новый текст.

2. **VaultViewModel.saveFile.**
   после `open(file)` (или напрямую) вызвать `saveFile(path, "новый")`; `advanceUntilIdle` →
   `repo.readFile(path) == "новый"` и `state.value.content == "новый"`.

## Приёмочные тест-кейсы (ручные, после разработки)
1. **Правка и сохранение**
   - Изначальное состояние: открыт файл `welcome.md` в просмотре.
   - Шаги: «Редактировать» → изменить текст → «Сохранить».
   - Ожидаемый результат: экран показывает отрендеренный обновлённый markdown; при повторном
     открытии файла изменения на месте (записаны на диск).
2. **Отмена правки**
   - Изначальное состояние: открыт файл, нажато «Редактировать», текст изменён.
   - Шаги: «Отмена».
   - Ожидаемый результат: возврат в просмотр со старым содержимым; файл на диске не изменён.
3. **Правка уходит в синк**
   - Изначальное состояние: файл отредактирован и сохранён; репозиторий настроен, вход выполнен.
   - Шаги: нажать «Синхронизировать» (или дождаться автосинка).
   - Ожидаемый результат: изменения закоммичены и запушены на сервер (видны в удалённом репо).
