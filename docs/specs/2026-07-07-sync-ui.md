# Спека: UI git-синка (кнопка + статус + диалог .md-конфликта) — слайс 3

Дата: 2026-07-07 · Tracker: none (key-less)
Опирается на: docs/specs/2026-07-07-git-sync-engine.md (движок `GitSync`/`JGitSync`).

## Цель
Проводка готового движока синка в приложение: кнопка «Синхронизировать», показ статуса и
диалог выбора версии при конфликте в `.md`-файле (реализация `ConflictResolver`). Синк
работает с той же папкой vault, что показывает вьювер.

## Границы (scope)
**В слайсе:** мост `UiConflictResolver` (suspend↔Compose), расширение `VaultViewModel`
(sync/статус/конфликт), кнопка+статус в UI, диалог выбора, конфиг репо через временные
константы (`BuildConfig` из `local.properties`), интеграция цели синка с папкой vault.
**НЕ в слайсе:** OAuth (реальное получение токена), автосинк (WorkManager), экран настроек
репозитория, diff-превью в диалоге (показываем только текст версий).

## Проектные решения
- **Интеграция с вьювером:** `SyncConfig.localPath` = папка vault, которую читает
  `VaultRepository`. Первый успешный синк = clone реального контента, затем список
  обновляется. **Авто-сид демо-файлов (`ensureSampleVault`) удаляется** — без настроенного
  репо vault пуст, экран показывает «Нет файлов» (поведение уже есть). Честнее для реального
  приложения: контент приходит из git.
- **Конфиг (временный):** `BuildConfig.SYNC_REMOTE_URL` и `BuildConfig.SYNC_TOKEN` из
  `local.properties` (токен не коммитится). Пустой URL → синк не зовёт движок, отдаёт
  `Failed("репозиторий не настроен")`. OAuth-слайс позже заменит только источник токена.
- **ViewModel:** расширяем существующий `VaultViewModel` (без координации нескольких VM).

## Архитектура и интерфейсы

### UiConflictResolver (commonMain, `app.obsidianmd.sync`) — чистая логика
```kotlin
class UiConflictResolver : ConflictResolver {
    private val _pending = MutableStateFlow<MdConflict?>(null)
    val pending: StateFlow<MdConflict?>          // != null → UI показывает диалог

    override suspend fun resolve(conflict: MdConflict): Resolution  // публикует pending, ждёт выбор
    fun choose(resolution: Resolution)            // вызывается из UI, разблокирует resolve
}
```
Движок на `.md`-конфликте зовёт `resolve()`; корутина ждёт `CompletableDeferred`, который
завершает `choose()`. Никакой зависимости от Compose — тестируется в `runTest`.

### VaultViewModel (commonMain, `app.obsidianmd.ui`) — расширение
Добавляемые члены (существующие files/selected/content/refresh/open/back не меняются):
```kotlin
sealed interface SyncStatus {
    data object Idle : SyncStatus
    data object Running : SyncStatus
    data class Done(val result: SyncResult) : SyncStatus
}
// новые зависимости конструктора: gitSync: GitSync, syncConfig: SyncConfig?, resolver: UiConflictResolver
val syncStatus: StateFlow<SyncStatus>
val pendingConflict: StateFlow<MdConflict?>   // = resolver.pending
fun sync()                                    // syncConfig==null/blank → Done(Failed(...)); иначе gitSync.sync → по успеху refresh()
fun resolveConflict(resolution: Resolution)   // = resolver.choose(resolution)
```

### Compose (commonMain, `app.obsidianmd.ui`)
- `VaultListScreen` — добавить действие «Синхронизировать» (иконка/кнопка) + строку статуса
  (Idle: скрыто; Running: «Синхронизация…»; Done(Cloned/UpToDate/Synced): краткий текст;
  Done(Failed): текст ошибки).
- `ConflictDialog(conflict, onChoose)` — показывается при `pendingConflict != null`; заголовок
  с путём файла, две кнопки «Оставить локальную» / «Взять серверную» → `onChoose(Resolution)`.
- `App` — рендерит диалог поверх при наличии конфликта.

### Android-проводка (androidMain)
- `composeApp/build.gradle.kts`: `buildConfigField` для `SYNC_REMOTE_URL`/`SYNC_TOKEN` из
  `local.properties` (пустые по умолчанию), включить `buildFeatures.buildConfig = true`.
- `MainActivity`: собрать `SyncConfig` (localPath = папка vault) из BuildConfig; пустой URL →
  передать `null`. Создать `UiConflictResolver`, `JGitSync`, прокинуть в `VaultViewModel`.
- Удалить вызов `ensureSampleVault` (и сам хелпер, если больше не нужен).

## Тестирование (TDD)

### commonTest (multiplatform, `runTest`)
1. **UiConflictResolver.**
   *Первый падающий тест:* в корутине вызвать `resolve(conflict)`; после `advanceUntilIdle`
   проверить `pending == conflict`; вызвать `choose(USE_LOCAL)`; проверить, что `resolve`
   вернул `USE_LOCAL` и `pending == null`. *Минимум:* StateFlow + CompletableDeferred.
2. **VaultViewModel.sync (фейковый `GitSync`).**
   - успех: фейк возвращает `Synced(...)` → `syncStatus` проходит `Running` → `Done(Synced)`,
     и после этого список файлов перечитан (`refresh` вызван — проверяем по обновлённому
     `state.files` через фейковый репозиторий).
   - пустой конфиг: `syncConfig == null` → `Done(Failed(...))`, фейк-движок НЕ вызван
     (флаг в фейке).
   - конфликт: фейк-движок внутри `sync` зовёт `resolver.resolve(...)` → `pendingConflict`
     становится не-null; `resolveConflict(USE_SERVER)` разблокирует, синк завершается
     `Done(Synced)`.
   *Первый падающий тест:* успех-кейс (нет `sync()`/`syncStatus` в VM).

## Приёмочные тест-кейсы (ручные, после разработки)
1. **Кнопка синка без конфигурации**
   - Изначальное состояние: свежий запуск, `SYNC_REMOTE_URL` пуст.
   - Шаги: нажать «Синхронизировать».
   - Ожидаемый результат: статус показывает ошибку «репозиторий не настроен», приложение не
     падает, список остаётся пустым.
2. **Успешный первый синк (clone)**
   - Изначальное состояние: в `local.properties` заданы `SYNC_REMOTE_URL` (тестовый GitHub с
     несколькими .md) и `SYNC_TOKEN` (PAT); vault пуст.
   - Шаги: нажать «Синхронизировать».
   - Ожидаемый результат: статус «Синхронизация…» → успех; список показывает файлы из репо;
     тап открывает отрендеренный markdown.
3. **Диалог выбора при .md-конфликте**
   - Изначальное состояние: файл `note.md` изменён и локально (в приложении/на диске), и на
     сервере (с другого клиента) по-разному; репо настроен.
   - Шаги: нажать «Синхронизировать»; в появившемся диалоге выбрать «Оставить локальную».
   - Ожидаемый результат: диалог показывает путь `note.md`; после выбора синк завершается,
     локальная версия сохранена (в списке/просмотре видно локальный вариант), пуш выполнен.
4. **Конфликт в не-.md → без диалога**
   - Изначальное состояние: не-.md файл (напр. `config.yml`) изменён и локально, и на сервере.
   - Шаги: нажать «Синхронизировать».
   - Ожидаемый результат: диалог НЕ появляется; синк завершается успехом; берётся серверная
     версия не-.md файла.
