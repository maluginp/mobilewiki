# Решение: стратегия локального git-хранения (shallow clone)

Тип: дизайн-решение (ADR) · Дата: 2026-07-06 · Tracker: none (key-less)
Статус: **принято** · Реализация: отложена (ляжет в будущий слайс git-синхронизации)

## Контекст
Android-приложение для Obsidian (KMP/Compose) синхронизирует markdown-vault через
GitHub. Опасение: git хранит полную историю, и на устройстве это может занимать много
места.

Уточнение предпосылки: для **текста** (markdown) история git очень компактна (zlib +
дельта-паки) — обычно единицы-десятки МБ за годы. Реальные пожиратели места —
**бинарные вложения** (картинки, PDF), которые не дельтятся, и длинная история с большим
churn. Отдельная законная задача — не тащить всю историю `.git` на телефон в принципе.

Продуктовое требование (согласовано): **история версий нужна только на сервере**; на
устройстве достаточно актуального состояния + возможности push/pull.

## Решение
Локальная копия ваулта — **shallow, single-branch**:

- Первичное клонирование: `git clone --depth=1 --single-branch --branch <main> <url>`.
- Последующая синхронизация: `git fetch --depth=1` (shallow-граница остаётся на глубине 1),
  затем merge/reset рабочего дерева и `git push`.
- Полная история коммитов существует **только на GitHub**. Клиент историю не хранит.

## Следствия
**Плюсы**
- `.git` на устройстве ≈ размер текущего ваулта. Раздувания из-за истории нет.
- Бинарные вложения хранятся только в **актуальной** версии → главный источник роста места
  устранён самим shallow-подходом (без git LFS).
- Меньше трафика и времени при первом клоне и при синке.

**Минусы / принятые компромиссы**
- На устройстве нет локального `git log` и отката к прошлым версиям заметок. По
  продуктовому решению — это не требуется (история доступна на GitHub через веб при
  необходимости).
- Shallow-специфика зависит от возможностей git-библиотеки на Android (см. Риск).

## Отвергнутые альтернативы
- **Полный clone** — на длинной истории и бинарниках `.git` растёт неограниченно. Именно
  та проблема, которую избегаем.
- **Partial / blobless clone** (`--filter=blob:none`) — блобы подтягиваются по требованию,
  история метаданных остаётся. Сложнее, оправдан только если понадобится доступ к истории
  на устройстве. Сейчас не нужен (история — серверная).
- **git LFS** — решает только большие бинарники, не историю, и добавляет требование к
  настройке на стороне репозитория. Shallow покрывает бинарники бесплатно.

## Риск для будущей реализации (валидировать спайком)
Перед полной реализацией слайса git-синка сделать короткий спайк на выбранной
git-библиотеке для Android (**кандидат по умолчанию — JGit**) и подтвердить:
1. `push` из shallow-клона работает против GitHub;
2. `fetch --depth=1` корректно сдвигает shallow-границу при последующих синках;
3. если JGit не тянет shallow-push — запасной вариант (полный clone одной ветки + внешняя
   обрезка истории, либо partial clone) переоценивается тогда же.

Спайк — гейт: если пункт 1/2 не подтверждается, стратегию пересматриваем до написания
основного кода синка.

## Границы (что НЕ входит)
Это решение описывает **только стратегию хранения**. Вне объёма и решается в отдельном
слайсе git-синхронизации:
- авторизация к GitHub (PAT / OAuth / device flow);
- поток commit / push / pull и его триггеры (ручной/авто);
- разрешение merge-конфликтов;
- выбор и интеграция конкретной git-библиотеки (спайк выше — часть того слайса).

## Проверка решения
Кода в этом заходе нет — валидируется не тестами, а спайком (см. Риск) в начале слайса
git-синка. Критерий принятия решения на практике: shallow-клон реального ваулта на
устройстве занимает ≈ размер текущего содержимого (не сумму всех версий), и push/pull
работают.
