# Решение: стратегия локального git-хранения

Тип: дизайн-решение (ADR) · Дата: 2026-07-06 · Пересмотрено: 2026-07-07 (после спайка)
Tracker: none (key-less)
Статус: **пересмотрено** — исходное решение (shallow clone) отменено находкой спайка;
действующее решение — **полный клон одной ветки** (см. «Пересмотр»).

## Контекст
Android-приложение для Obsidian (KMP/Compose) синхронизирует markdown-vault через
GitHub. Опасение: git хранит полную историю, и на устройстве это может занимать много
места.

Уточнение предпосылки: для **текста** (markdown) история git очень компактна (zlib +
дельта-паки) — обычно единицы-десятки МБ за годы. Реальные пожиратели места —
**бинарные вложения** (картинки, PDF), которые не дельтятся, и длинная история с большим
churn. Продуктовое требование (исходно): история версий нужна только на сервере.

## Исходное решение (2026-07-06) — ОТМЕНЕНО
Shallow single-branch clone (`--depth=1`, далее `fetch --depth=1`); история — только на
сервере. Риск, вынесенный тогда же на спайк: подтвердить, что JGit умеет push из shallow
и корректно двигает shallow-границу.

## Пересмотр (2026-07-07): результат спайка
Спайк проведён в рамках слайса движка синка (JGit, тесты против локального bare-репо):

- **Push из shallow-клона — работает** (тест commit+push зелёный).
- **Но 3-way merge на shallow-клоне теряет данные.** JGit-merge требует общий commit-предок
  (merge-base); `--depth=1` обрезает историю до одного коммита, merge-base недоступен, и при
  одновременных правках merge затирает неконфликтующие локальные изменения серверной
  версией. Воспроизводится детерминированно; на полном клоне тот же сценарий проходит.

Вывод: shallow-стратегия несовместима с корректным клиентским merge в JGit. Экономия для
**текста** была изначально невелика (см. «Контекст»), а shallow всё равно **не** решает
главный источник роста — историю **бинарников** (в покое текущие версии хранятся в обоих
случаях). Цена отказа от shallow — низкая; цена сохранения — потеря данных при синке.

## Действующее решение (2026-07-07)
Локальная копия — **полный клон одной ветки**:

- Клонирование: `clone --single-branch --branch <main>` (без `--depth`).
- Синхронизация: `fetch` + commit локальных правок + merge («сервер выигрывает» на
  конфликтах) + push. merge-base доступен → слияние корректно.
- Хранится история одной ветки. Для текстового ваулта это единицы-десятки МБ — приемлемо.

## Следствия
**Плюсы**
- Корректный merge и стратегия «сервер выигрывает» без потери неконфликтующих правок.
- Проще код (нет shallow-специфики и её краевых случаев).
- Доступна локальная история одной ветки (бонус, хоть продуктом и не требовалась).

**Минусы / принятые компромиссы**
- Локально хранится история ветки. Для текста мало; **для бинарных вложений может расти** —
  выносится в отдельную будущую задачу (ниже).

## Отдельная будущая задача — бинарные вложения
Если ваулт содержит крупные бинарники (картинки/PDF) и их история раздувает `.git` —
решать **отдельно** и независимо от стратегии клона: git LFS, либо исключение вложений из
git-синка, либо `sparse-checkout`. Shallow эту проблему не решал, поэтому его отмена ничего
здесь не ухудшает.

## Отвергнутые альтернативы
- **Shallow `--depth=1`** — отменено спайком: ломает клиентский merge (потеря данных).
- **Shallow + deepen-on-merge** (до-fetch истории до нахождения merge-base) — держит диск
  малым, но сложнее и не на 100% предсказуемо по требуемой глубине; для текста экономия не
  оправдывает сложность.
- **Partial / blobless clone** (`--filter=blob:none`) — сложнее, оправдан только при нужде в
  истории на устройстве по требованию; не нужен.
- **git LFS** — решает только бинарники, не общую стратегию клона; см. «будущая задача».

## Влияние на слайс движка синка
Слайс docs/specs/2026-07-07-git-sync-engine.md обновить: убрать `setDepth(1)` из clone и
fetch; тест, проверявший `.git/shallow`, заменить на проверку корректного полного клона
одной ветки. Остальная логика (commit/fetch/merge/server-wins/push) не меняется — она уже
зелёная на полном клоне.
