# Спека: автосинк на WorkManager — слайс 6

Дата: 2026-07-08 · Tracker: none (key-less)
Опирается на: движок/UI синка, OAuth, настройки (docs/specs/2026-07-07-*).

## Цель
Периодически синхронизировать vault в фоне поверх готового движка `GitSync`, используя URL
из `RepoSettingsStore` и токен из `TokenStore`. В фоне диалога нет → `.md`-конфликты решаются
«локальная побеждает» (USE_LOCAL).

## Границы (scope)
**В слайсе:** `BackgroundSyncRunner` (ядро), `SyncWorker` (CoroutineWorker), `AutoSyncScheduler`
(периодическая задача), планирование при входе, зависимость WorkManager.
**НЕ в слайсе:** интервал в настройках (фиксированный ~30 мин), кнопка logout/cancel, синк при
изменении файла (редактирования ещё нет), нотификации о результате.

## Проектные решения
- **Конфликт в фоне:** `.md` → `USE_LOCAL` (локальные правки не теряются; серверная версия
  остаётся в истории git). Не-`.md` — как в движке (сервер).
- **Частота:** `PeriodicWorkRequest` ~30 мин, `NetworkType.CONNECTED`, уникальная задача
  (`ExistingPeriodicWorkPolicy.KEEP`). Планируется при входе (`loggedIn`).
- **Гард:** если URL не задан — воркер ничего не делает (runner отдаёт `Failed`), не падает.

## Архитектура и интерфейсы

### BackgroundSyncRunner (commonMain, `app.obsidianmd.sync`) — тестируемое ядро
```kotlin
class BackgroundSyncRunner(
    private val gitSync: GitSync,
    private val syncConfigProvider: () -> SyncConfig?,
) {
    private val localWins = ConflictResolver { Resolution.USE_LOCAL }
    suspend fun run(): SyncResult {
        val cfg = syncConfigProvider() ?: return SyncResult.Failed("репозиторий не настроен")
        return gitSync.sync(cfg, localWins)
    }
}
```
Без UI, без refresh — только синк с резолвером «локальная побеждает».

### SyncWorker (androidMain) — тонкая обёртка
`class SyncWorker(context, params) : CoroutineWorker(context, params)`:
- собирает `EncryptedTokenStore`, `SharedPrefsRepoSettingsStore`, `JGitSync`, путь vault
  (`context.filesDir/vault`);
- `syncConfigProvider = { settingsStore.getRemoteUrl()?.takeIf { it.isNotBlank() }?.let {
  SyncConfig(remoteUrl = it, localPath = vaultPath, token = tokenStore.get()) } }`;
- `doWork()`: `when (BackgroundSyncRunner(JGitSync(), provider).run()) { Failed -> Result.retry(); else -> Result.success() }`.

### AutoSyncScheduler (androidMain)
```kotlin
class AutoSyncScheduler(context) {
    fun schedule()  // enqueueUniquePeriodicWork("obsidian-autosync", KEEP, PeriodicWorkRequest<SyncWorker>(30, MINUTES) + NetworkType.CONNECTED)
    fun cancel()    // cancelUniqueWork("obsidian-autosync")
}
```

### Проводка (MainActivity)
При `loggedIn` вызвать `AutoSyncScheduler(applicationContext).schedule()` (один раз при старте).

### Зависимость
`androidx.work:work-runtime-ktx` (в androidMain).

## Тестирование (TDD)

### commonTest (`runTest`)
`BackgroundSyncRunnerTest` на фейках:
1. **Нет конфига.** *Первый падающий тест:* `syncConfigProvider = { null }` → `run()` возвращает
   `Failed`, фейк-движок не вызван (флаг `called`). *Минимум:* класс + гард.
2. **С конфигом.** провайдер отдаёт `SyncConfig`; фейк возвращает `Synced(...)` → `run()` вернул
   его, `called == true`.
3. **`.md`-конфликт → USE_LOCAL.** фейк-движок внутри `sync` зовёт
   `resolver.resolve(MdConflict("n.md","L","S"))` и кодирует выбор в результат
   (`conflictsResolved = if (USE_LOCAL) 1 else 0`) → `run()` даёт `conflictsResolved == 1`.

## Приёмочные тест-кейсы (ручные, после разработки)
1. **Фоновый синк обновляет vault**
   - Изначальное состояние: вход выполнен, URL репо задан, есть сеть.
   - Шаги: изменить файл на сервере (с другого клиента); подождать интервал (или принудительно
     запустить задачу через `adb`/WorkManager-инспектор).
   - Ожидаемый результат: локальный vault обновился без ручного нажатия «Синхронизировать».
2. **Конфликт в фоне → локальная версия**
   - Изначальное состояние: один `.md` изменён и локально, и на сервере; настроен репо.
   - Шаги: дождаться фонового синка.
   - Ожидаемый результат: локальная версия файла сохранена (USE_LOCAL); приложение/воркер не
     падает; серверная версия доступна в истории git.
3. **Без URL — воркер не падает**
   - Изначальное состояние: вход выполнен, URL репо не задан.
   - Шаги: дождаться запуска задачи (или запустить принудительно).
   - Ожидаемый результат: задача завершается без краха, vault не меняется.
