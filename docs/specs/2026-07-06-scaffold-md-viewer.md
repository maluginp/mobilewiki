# Спека: скаффолд KMP/Compose + просмотр md-файлов

Источник: docs/tasks/2026-07-06-scaffold-md-viewer.md · Tracker: none (key-less)

## Цель
Заложить основу Android-приложения для Obsidian (KMP + Compose Multiplatform + корутины) и
дать просмотр локальных markdown-файлов: список .md в vault-папке → рендер выбранного файла.

## Границы (scope)
**В этом слайсе:** скаффолд Gradle-проекта, экран списка md-файлов, экран рендера markdown,
чтение файлов вне UI-потока, тесты на логику листинга.
**НЕ в этом слайсе:** git-синхронизация, редактирование, поиск, AI/OpenRouter, выбор vault
через системный picker (папка пока задаётся дефолтом/константой). Это следующие слайсы.

## Проектные решения (дефолты)
- **Структура:** один Gradle-проект, модуль `composeApp` с Kotlin Multiplatform плагином,
  таргет `androidTarget()`. Общий код в `commonMain`, Android-точка входа в `androidMain`.
  KMP-структура заложена сразу, но собираем пока только Android.
- **UI:** Compose Multiplatform (`org.jetbrains.compose`), Material3.
- **Markdown-рендер:** готовая библиотека `com.mikepenz:multiplatform-markdown-renderer`
  (Compose Multiplatform) — не пишем свой парсер. ponytail: реюз зависимости.
- **Файлы:** vault-папка = `context.filesDir/vault` (создаётся с 1–2 семпл-файлами при
  первом запуске, чтобы было что показать). Листинг/чтение — через `okio` FileSystem в
  `commonMain`, что делает логику листинга чистой и тестируемой.
- **Навигация:** без навигационной библиотеки — простое состояние
  `selectedFile: String?` в Compose (список ⇄ просмотр). ponytail: nav-либа не нужна для
  двух экранов.
- **Асинхронность:** чтение файла в `viewModelScope`/корутине на `Dispatchers.IO`.

## Функциональные требования
1. Проект собирается (`:composeApp:assembleDebug`) и запускается на Android.
2. Экран списка: показывает все `*.md` в vault, отсортированные по имени; пустой vault →
   заметное «нет файлов».
3. Тап по файлу → экран просмотра с отрендеренным markdown (заголовки, списки,
   bold/italic, ссылки, инлайн/блок-код). Кнопка «назад» возвращает к списку.
4. Чтение содержимого не блокирует UI.

## Компоненты
- `VaultRepository` (commonMain): `listMarkdownFiles(): List<MdFile>`,
  `readFile(path): String`. Инкапсулирует okio FileSystem (внедряется для тестов).
- `MdFile(name, path)` — модель.
- `VaultViewModel`: состояние (`files`, `selected`, `content`, `loading`), корутины.
- Compose: `VaultListScreen`, `MarkdownScreen`, `App` (переключение по `selected`).

## Приёмочные тест-кейсы (ручные, до мержа)
1. **Список файлов**
   - Изначальное состояние: свежая установка, vault с 2 семпл-файлами (`welcome.md`,
     `notes.md`).
   - Шаги: запустить приложение.
   - Ожидаемый результат: на первом экране список из двух пунктов, отсортированных по имени.
2. **Рендер markdown**
   - Изначальное состояние: открыт список файлов.
   - Шаги: тап по `welcome.md` (содержит заголовок, список, **bold**, ссылку, `code`).
   - Ожидаемый результат: открывается экран просмотра; markdown отрендерен (заголовок
     крупнее, список с маркерами, жирный текст, кликабельная ссылка, моноширинный код), а не
     показан как сырой текст.
3. **Назад к списку**
   - Изначальное состояние: открыт экран просмотра файла.
   - Шаги: нажать «назад».
   - Ожидаемый результат: возврат к списку файлов, приложение не падает.
4. **Пустой vault**
   - Изначальное состояние: vault-папка пуста.
   - Шаги: запустить приложение.
   - Ожидаемый результат: экран со списком показывает сообщение «нет файлов», без краша.

## Автотесты (TDD, не-UI)
- `VaultRepository.listMarkdownFiles` на in-memory okio FileSystem: фильтрует только `.md`,
  игнорирует прочие расширения и подкаталоги-шум, сортирует по имени, пустая папка → пустой
  список.
- `readFile` возвращает содержимое существующего файла.
