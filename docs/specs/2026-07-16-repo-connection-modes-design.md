# Спека: три режима подключения репозитория

## Проблема

Публикацию в сторах заблокировали: единственный способ входа — GitHub OAuth, требуется
альтернативный способ. Нужно дать пользователю несколько способов подключить хранилище заметок,
не завязанных на GitHub-аккаунт, и объяснить, что вообще делает приложение.

## Цель

Онбординг и настройки поддерживают **три режима** подключения хранилища:

1. **GitHub** — вход через OAuth (как сейчас), затем выбор репозитория из списка.
2. **Любой git-репозиторий** — ввод URL + токен/пароль, синхронизация по HTTPS с любым хостом
   (GitHub / GitLab / Bitbucket / self-hosted).
3. **Локальное хранилище** — без удалённого репозитория и без синхронизации.

Наличие режима 2 (вход без GitHub-аккаунта) закрывает требование сторов об альтернативном входе.

## Ключевой факт (почему это дёшево)

Слой git уже провайдер-независимый: `SyncConfigProvider`
(`composeApp/.../di/AppModule.kt`) собирает `SyncConfig(remoteUrl, token, localPath)` и клонит
по HTTPS на любой хост. `TokenStore` (шифрованный на Android) и `RepoSettingsStore` уже есть.
Основной новый код — точки входа в UI, сохранение введённого токена в те же хранилища и
host-agnostic проверка доступа.

## Границы (вне итерации)

- **SSH-доступ** — генерация/менеджмент ключей на устройстве. Отложено.
- **Мульти-репозиторий** — один активный репозиторий и один слот токена, как сейчас.
- **Очистка/переклон при смене репозитория** — существующее поведение синка не меняем, только
  предупреждаем пользователя о возможной потере несинхронизированных заметок.

## Архитектура и компоненты

### A. Онбординг

**`WelcomeScreen`** (`features/onboarding/impl/.../presentation/WelcomeScreen.kt`)
- Объяснение, что делает приложение (заметки в Markdown + синхронизация через git).
- Три кнопки-действия:
  - «Войти через GitHub» → `onSignInGitHub` (существующий OAuth device-flow → RepoPicker).
  - «Подключить любой git-репозиторий» → `onConnectByUrl` → шаг `ManualUrl`.
  - «Использовать локально (без синхронизации)» → `onUseLocal` → помечаем онбординг завершённым
    и `onFinished()` без установки remote.

**`ManualUrlScreen`** (`.../presentation/ManualUrlScreen.kt`)
- Поле URL.
- Необязательное `secret`-поле «Токен доступа / пароль» (show/hide, как `SettingField(secret=true)`
  в настройках).
- Если поле токена пустое → предупреждение: «Без токена доступа изменения не будут сохраняться в
  репозиторий — только чтение».
- `onSubmit(url, token)`.

**`ManualConnectViewModel`** (новый, `.../onboarding/ManualConnectViewModel.kt`)
- `connect(url: String, token: String)`: если `token` непустой → `tokenStore.save(token)`; отдаёт
  `url` для перехода на шаг `Validate`.
- Единственная логика с ветвлением в части A → покрывается юнит-тестом (`FakeTokenStore` уже есть).

### B. Проверка доступа (шаг `Validate`)

Проверка доступа становится **host-agnostic** — реальный `git ls-remote` с учётными данными,
вместо прежней GitHub-only проверки через `api.github.com`.

**`RepoAccessCheck`** — новый порт в `:features:sync:api`
```kotlin
interface RepoAccessCheck {
    suspend fun check(url: String, token: String?): AccessResult
}
```
- `AccessResult` (переносится/переиспользуется): `Ok` | `Denied(reason)` | `Unknown(reason)`.
- Реализация на JGit (рядом с `JGitSync`): `Git.lsRemoteRepository().setRemote(url)
  .setCredentialsProvider(<token>).call()` на IO-диспетчере.
  - Успех → `Ok`.
  - Ошибка транспорта/авторизации (`TransportException`) → `Denied`.
  - Прочее/сеть → `Unknown`.
- Токен передаётся как пароль HTTPS-учётки (для GitHub/GitLab PAT работает с любым username).

**`RepoValidationViewModel`** (`.../onboarding/RepoValidationViewModel.kt`)
- Переключается с `RepoAccess` на `RepoAccessCheck`.
- **С экрана `ManualUrl` нельзя пройти дальше без `Ok`**: при `Denied`/`Unknown` показывается
  ошибка, пользователь остаётся на валидации (retry). Токен уже сохранён к моменту проверки —
  `ls-remote` его использует.
- Публичный репозиторий без токена → `ls-remote` без учётки → `Ok`.

`GitHubRepoAccess` (ktor-проверка) после переключения не нужен для этого пути — если он больше
нигде не используется, удаляется вместе с неиспользуемыми зависимостями; иначе OAuth-путь
остаётся как есть.

### C. Завершение онбординга (для локального режима)

Сейчас «онбординг завершён» выводится из наличия токена/URL. У локального режима нет ни того, ни
другого, поэтому вводится явный флаг.

**`RepoSettingsStore`** (`:features:settings:api` + `SharedPrefsRepoSettingsStore`)
- Добавляются `getOnboardingDone(): Boolean` / `setOnboardingDone(done: Boolean)` (persist в
  SharedPrefs, ключ `onboarding_done`).
- Ставится `true` при финише **любого** режима.

**`startStack`** (`composeApp/.../nav/StartStack.kt`) — новая сигнатура
`startStack(onboardingDone: Boolean, hasToken: Boolean)`:
- `onboardingDone` → `VaultList`.
- `!onboardingDone && hasToken` → `Onboarding(RepoPicker)` (возобновление GitHub после рестарта).
- `!onboardingDone && !hasToken` → `Onboarding(Login)` (Welcome с тремя режимами).

Точка вызова `startStack` обновляется под новую сигнатуру (передаёт `getOnboardingDone()` вместо
вычисленного `hasRepo`).

### D. Настройки — управление репозиторием

**`SettingsScreen`** (`features/settings/impl/.../presentation/SettingsScreen.kt`), секция
репозитория:
- Показ **текущего режима**: если `remoteUrl` задан — сам URL; иначе «Локальное хранилище (без
  синхронизации)».
- Кнопка **«Изменить репозиторий»** → сначала диалог-предупреждение: «Будут показаны заметки
  нового репозитория. Несинхронизированные заметки могут быть потеряны.» → при подтверждении
  показывается выбор режима (GitHub / Вручную / Локально), дальше — как в онбординге.
- Инлайновое редактирование URL + «Сохранить» убирается (заменяется flow «Изменить»).
- «Синхронизировать сейчас» остаётся без изменений.

**`OnboardingStart`** (`:features:onboarding:api`)
- Добавляется значение `ManualUrl`; в `startStep` — маппинг `ManualUrl -> Step.ManualUrl`, чтобы
  настройки могли открыть ручной ввод напрямую.
- GitHub-роутинг (токен → RepoPicker, иначе Login) остаётся в nav-слое; фича настроек дёргает
  колбэки (`onConnectManually`, `onPickFromGitHub`, `onUseLocal`).

## Поток данных

1. **GitHub**: Welcome → OAuth (`AuthViewModel` сохраняет токен) → RepoPicker (список реп по
   токену) → Validate (`ls-remote` ok) → `setRemoteUrl(url)` + `setOnboardingDone(true)` → Vault.
2. **Любой git**: Welcome → ManualUrl (URL + токен) → `connect` (сохранение токена) → Validate
   (`ls-remote` с токеном) → `setRemoteUrl(url)` + `setOnboardingDone(true)` → Vault.
3. **Локально**: Welcome → «Использовать локально» → `setOnboardingDone(true)` → Vault; remote не
   задан, `SyncConfigProvider` возвращает `null`, синка нет.
4. **Смена из настроек**: Settings → «Изменить» → предупреждение → выбор режима → соответствующий
   flow 1–3 поверх списка заметок.

## Обработка ошибок

- Валидация доступа: `Denied` → сообщение «Нет доступа к репозиторию, проверьте ссылку и токен»,
  остаёмся на шаге; `Unknown` → сообщение о сетевой ошибке + retry.
- OAuth-ошибки — существующее поведение `AuthState.Failed` (retry).
- Пустой токен на приватном репо → `ls-remote` вернёт `Denied` → пользователь увидит ошибку и
  добавит токен.

## Тестирование (TDD)

Для каждого юнита — сначала падающий тест, затем минимальная реализация.

### `ManualConnectViewModel`
- **Первый падающий тест:** класса/метода ещё нет. `connect("https://gitlab.com/x/y.git",
  "glpat-abc")` → `FakeTokenStore.get() == "glpat-abc"`, состояние отдаёт url для перехода.
- **Минимальный код:** метод сохраняет непустой токен и прокидывает url.
- **Второй тест:** `connect(url, "")` → `save` **не** вызывается (публичный репозиторий не
  затирает существующий токен).
- **Изоляция:** зависит только от `TokenStore` (фейк уже есть), без сети и навигации.

### `startStack`
- **Первый падающий тест:** новая сигнатура `startStack(onboardingDone, hasToken)`.
  - `startStack(onboardingDone = true, hasToken = <любой>)` → `[VaultList]`.
  - `startStack(false, hasToken = true)` → `[Onboarding(RepoPicker)]`.
  - `startStack(false, hasToken = false)` → `[Onboarding(Login)]`.
- **Минимальный код:** `when` по флагу завершения и токену.
- **Изоляция:** чистая функция, без зависимостей.

### `startStep`
- **Тест:** `startStep(OnboardingStart.ManualUrl) == Step.ManualUrl` (плюс существующие Login/
  RepoPicker остаются).
- **Изоляция:** чистая функция.

### `RepoValidationViewModel`
- **Тест:** с фейковым `RepoAccessCheck`, возвращающим `Ok` → состояние `ValidationState.Ok`;
  `Denied` → `ValidationState.Denied` (не пропускает дальше). Фейк подменяет сеть.
- **Изоляция:** зависит от интерфейса `RepoAccessCheck`, реальный JGit-чек не участвует в юните.

### Вне юнит-тестов (composable / интеграция)
- `WelcomeScreen` (3 кнопки + текст), предупреждение read-only, диалог выбора режима, диалог
  предупреждения о смене репо — проверяются в acceptance.
- JGit-реализация `RepoAccessCheck` (`ls-remote`) — проверяется в acceptance (реальная сеть).

## Приёмочные тест-кейсы (ручной прогон после разработки)

### 1. Вход через GitHub не сломан
- **Изначальное состояние:** свежая установка, экран Welcome.
- **Шаги:** «Войти через GitHub» → пройти device-flow → выбрать репозиторий из списка.
- **Ожидаемый результат:** онбординг завершён, открывается список заметок, синхронизация работает.

### 2. Вход по приватной git-ссылке с токеном
- **Изначальное состояние:** свежая установка, экран Welcome; есть приватный репозиторий и
  валидный токен/PAT.
- **Шаги:** «Подключить любой git-репозиторий» → ввести URL приватного репо и токен → Continue.
- **Ожидаемый результат:** проверка доступа проходит (`Ok`), репозиторий склонирован, открывается
  список заметок.

### 3. Приватная ссылка с неверным токеном
- **Изначальное состояние:** свежая установка, экран Welcome.
- **Шаги:** «Подключить любой git-репозиторий» → ввести URL приватного репо и заведомо неверный
  токен → Continue.
- **Ожидаемый результат:** показана ошибка доступа, дальше не пускает; после ввода верного токена
  и повтора — проходит.

### 4. Публичная ссылка без токена
- **Изначальное состояние:** свежая установка, экран Welcome; есть публичный репозиторий.
- **Шаги:** «Подключить любой git-репозиторий» → ввести URL публичного репо, поле токена оставить
  пустым.
- **Ожидаемый результат:** под полем показано предупреждение об режиме «только чтение»; проверка
  доступа проходит; репозиторий склонирован.

### 5. Локальное хранилище без синхронизации
- **Изначальное состояние:** свежая установка, экран Welcome.
- **Шаги:** «Использовать локально (без синхронизации)».
- **Ожидаемый результат:** приложение открывается с локальным хранилищем; кнопка синхронизации не
  выполняет удалённых операций (remote не задан); после перезапуска приложение сразу открывает
  заметки, а не онбординг.

### 6. Настройки: просмотр текущего режима
- **Изначальное состояние:** онбординг пройден (любой режим), открыты настройки.
- **Шаги:** посмотреть секцию репозитория.
- **Ожидаемый результат:** для git-режимов показан URL, для локального — «Локальное хранилище (без
  синхронизации)».

### 7. Настройки: смена репозитория с предупреждением
- **Изначальное состояние:** онбординг пройден, открыты настройки.
- **Шаги:** «Изменить репозиторий» → прочитать предупреждение → подтвердить → выбрать режим →
  завершить подключение (например, «Вручную» с новым URL+токеном).
- **Ожидаемый результат:** показано предупреждение о возможной потере несинхронизированных
  заметок; после подтверждения и подключения активный репозиторий сменился, показаны заметки
  нового репозитория.
