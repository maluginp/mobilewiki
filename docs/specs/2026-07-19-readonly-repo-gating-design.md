# Дизайн: блокировка редактирования для read-only репозитория

Дата: 2026-07-19 · Задача: [docs/tasks/2026-07-19-readonly-repo-gating.md](../tasks/2026-07-19-readonly-repo-gating.md)
Tracker: none (free-text) · Flow: mr (GitHub remote → MR отложен на ручной PR)

## Проблема

Приложение позволяет создавать заметки/папки и сохранять правки даже когда у пользователя нет
права записи (push) в подключённый git-репозиторий. Такие изменения делаются локально, но при
синхронизации push отклоняется — работа теряется впустую. Нужно на экране проверки доступа
определять право записи и, если репозиторий доступен только на чтение, заблокировать
редактирование/добавление, оставив просмотр и AI.

## Ключевое решение: как определять право записи

**Выбрано (по умолчанию): host-agnostic проба push-соединения.** После успешного `lsRemote`
(подтверждает чтение) пробуем открыть push-соединение к `git-receive-pack` (advertisement refs),
**ничего не отправляя**. GitHub/GitLab по HTTPS отдают receive-pack только пользователям с правом
записи — read-only получает auth-ошибку (401/403 → `TransportException`). Проба ничего не мутирует.

- Плюсы: работает для любого git-хоста (не только GitHub, не только выбор из списка), живёт ровно
  на экране проверки доступа, переиспользует уже имеющийся `RepoAccessCheck`.
- Минусы: один дополнительный сетевой round-trip на экране проверки (приемлемо — там уже есть
  `lsRemote`).

**Отвергнутая альтернатива:** брать `permissions.push` из ответа GitHub API `/user/repos` в момент
выбора репозитория. Проще по коду, но работает только для GitHub-репозиториев, выбранных из списка,
и не для ручного ввода URL / GitLab. Не кладётся на «проверку на экране доступа». Поэтому — проба.

## Архитектура

Поток данных: **проверка доступа (проба записи) → персист флага writable → гейтинг UI по флагу.**

### 1. Проба записи в проверке доступа

`AccessResult.Ok` перестаёт быть `data object` и несёт результат пробы:
```
data class Ok(val canWrite: Boolean) : AccessResult
```
`Denied`/`Unknown` — без изменений.

`JGitRepoAccessCheck.check(url, token)`: после успешного `lsRemote` (чтение подтверждено) —
проба записи:
```
val canWrite = try {
    val t = Transport.open(URIish(url))
    token?.takeIf { it.isNotBlank() }?.let {
        t.credentialsProvider = UsernamePasswordCredentialsProvider(it, "")
    }
    val push = t.openPush(); push.close(); t.close()   // contacts git-receive-pack, ничего не пушим
    true
} catch (e: Exception) { false }
AccessResult.Ok(canWrite)
```
Если `lsRemote` упал — прежнее поведение (`Denied`/`Unknown`), пробы записи нет.

### 2. Проброс через VM проверки

`ValidationState.Ok` → `data class Ok(val canWrite: Boolean)`. `RepoValidationViewModel.validate`
мапит `AccessResult.Ok(canWrite)` → `ValidationState.Ok(canWrite)`.

`RepoValidationScreen` (ветка Ok) показывает разный текст: полный доступ vs «только чтение —
редактирование будет недоступно» (новые строки локализации). Кнопка «Продолжить» — как сейчас.

### 3. Персист флага

`RepoSettingsStore` получает:
```
fun getWritable(): Boolean       // default true
fun setWritable(writable: Boolean)
```
Дефолт `true` — чтобы локальный режим и уже существующие установки не блокировались. Реализация в
`SharedPrefsRepoSettingsStore` (bool в SharedPrefs), плюс `FakeRepoSettingsStore` для тестов.

Запись флага — в `OnboardingPresentationProviderImpl`, шаг `Validate`, в `onContinue`:
```
onContinue = {
    settings.setRemoteUrl(key.url)
    settings.setWritable((state as? ValidationState.Ok)?.canWrite ?: true)
    settings.setOnboardingDone(true)
    onFinished()
}
```
Локальный режим (`onUseLocal`) и ручные пути ничего не пишут → остаётся дефолт `true`. Смена
репозитория проходит через тот же шаг `Validate` → флаг обновляется автоматически.

### 4. Гейтинг UI по флагу

Флаг читается **один раз** в `AppNavHost` через `koinInject<RepoSettingsStore>()`:
`val readOnly = !settings.getWritable()`. Реактивный flow не нужен — единственный момент смены
доступа (онбординг/смена репо) заканчивается `onFinished()` → `resetTo(VaultList)` → рекомпозиция
перечитывает флаг. (ponytail: без реактивности, апгрейд — сделать `getWritable` потоком, если
понадобится менять доступ на лету.)

Гейтинг — прокидываем `readOnly: Boolean` (или инвертированный `canEdit`) в два экрана:
- **`ListScreen`** (`VaultPresentationProvider` + impl + `VaultListScreen`): при `readOnly` **не
  показывать FAB** создания заметки/папки. `onCreateNote`/`onCreateFolder` остаются в сигнатуре, но
  FAB скрыт.
- **`NoteScreen`** (`NotePresentationProvider` + impl + `NoteScreenContent`): при `readOnly` —
  режим только чтения: иконка редактирования/сохранения скрыта, вход в редактор недоступен.
- **AI не трогаем** — по явному требованию. Нижняя навигация Brain↔AI и AI-чат работают как есть.

## Обработка ошибок

- Проба записи бросает исключение → трактуем как **нет записи** (`canWrite = false`). Консервативно:
  лучше показать read-only, чем разрешить правки, которые не за-push-атся. `lsRemote` при этом уже
  прошёл, так что чтение точно есть.
- `Denied`/`Unknown` от `lsRemote` — поток не доходит до сохранения флага (пользователь не проходит
  дальше экрана проверки), гейтинг не задействован.
- Флаг отсутствует в хранилище (старая установка) → `getWritable()` возвращает `true` → поведение
  как раньше. Обратная совместимость.

## Тестирование (TDD — для каждой единицы первый падающий тест)

1. **`RepoValidationViewModel` (маппинг canWrite).** Первый падающий тест:
   `validate_ok_readonly_maps_canWrite_false` — `FakeAccessCheck(AccessResult.Ok(canWrite=false))`
   → `state is ValidationState.Ok && !state.canWrite`. Минимальный код: `Ok` несёт `canWrite`, VM
   пробрасывает. Плюс обновить существующий `validate_ok` под `Ok(true)`. Изоляция: fake
   `RepoAccessCheck`, уже есть.
2. **`RepoSettingsStore` контракт (writable).** Первый падающий тест:
   `writable_defaults_true_and_persists` — свежий стор `getWritable()==true`; после
   `setWritable(false)` → `false`. Минимальный код: пара геттер/сеттер в fake и SharedPrefs-импле.
   Изоляция: `RepoSettingsStoreContractTest` (общий контракт) + `FakeRepoSettingsStore`.
3. **`JGitRepoAccessCheck` (happy-path записи).** Первый падающий тест:
   `writable_bare_repo_reports_canWrite_true` — против реального локального bare-репо (`BareRepo.kt`,
   file-транспорт пишущий) `check()` → `Ok(canWrite=true)`. Минимальный код: проба `openPush`.
   Ветку `canWrite=false` на локальном file-репо смоделировать нельзя (нет auth) — её покрывает
   маппинг-тест VM (#1); в спеке это зафиксировано как осознанное ограничение unit-уровня.
4. **`VaultListScreen` (скрытие FAB).** Первый падающий тест: `readOnly_hides_create_fab` —
   рендер с `readOnly=true` → FAB создания отсутствует; с `readOnly=false` → присутствует.
   Харнес: существующий `VaultListScreenTest` (Robolectric + compose uiTest).
5. **`NoteScreen` (только чтение).** Первый падающий тест: `readOnly_hides_edit_action` — рендер с
   `readOnly=true` → иконки редактирования/сохранения нет. Харнес: compose uiTest для note.

Все затронутые модули должны оставаться зелёными (полный прогон `:test`).

## Приёмочные тест-кейсы (ручной прогон после разработки)

**1. Read-only репозиторий: редактирование заблокировано, просмотр и AI работают**
- Изначальное состояние: приложение установлено, пользователь авторизован в GitHub, выбирает
  репозиторий, к которому у него доступ **только на чтение** (например, чужой публичный репозиторий).
- Шаги: пройти онбординг до экрана проверки доступа → дождаться результата → «Продолжить» → на
  списке файлов посмотреть наличие кнопки «+»; открыть любую заметку; открыть AI-чат.
- Ожидаемый результат: экран проверки сообщает про доступ только на чтение; на списке **нет** кнопки
  «+» (создание заметки/папки недоступно); заметка открывается, но **нет** кнопки редактирования/
  сохранения; AI-чат открывается и отвечает.

**2. Read-write репозиторий: всё доступно (регресс)**
- Изначальное состояние: пользователь авторизован, выбирает **свой** репозиторий с правом записи.
- Шаги: пройти проверку доступа → «Продолжить» → на списке нажать «+», создать заметку, ввести текст,
  сохранить; создать папку.
- Ожидаемый результат: экран проверки — полный доступ; кнопка «+» есть; заметка создаётся и
  сохраняется; папка появляется в списке. Поведение как до изменения.

**3. Локальный режим: запись доступна**
- Изначальное состояние: на приветственном экране выбран локальный режим (без git-remote).
- Шаги: открыть список → «+» → создать заметку → сохранить.
- Ожидаемый результат: создание и сохранение работают (локальный режим всегда на запись).

**4. Совместимость со старой установкой**
- Изначальное состояние: приложение обновлено с версии без флага writable (в хранилище флага нет),
  репозиторий уже подключён.
- Шаги: открыть список и заметку.
- Ожидаемый результат: редактирование/создание доступны (дефолт writable=true), приложение не
  блокирует работу до следующей проверки доступа.
```
